# v3.2 安全性閉環 — 新舊版差異與 In-Circuit / Off-Chain 取捨

**日期**：2026-04-28
**範圍**：sdjwt_adapter、jwt_x5c_adapter、benchmark/spec-check
**動因**：審計指出 v3 的 in-circuit 證明與 issuer-signed bytes 之間缺少最後一道綁定，導致 prover 可以對 commit 與被簽資料做 mix-and-match。

---

## 0. TL;DR

| 項目 | v3 / v3.1 (舊) | v3.2 (新) |
|---|---|---|
| **SD-JWT** disclosure 是否真在 issuer-signed payload 裡 | ❌ 只有 Pedersen disclosure_root，需 verifier 在 off-chain 對 manifest | ✅ in-circuit base64url 編碼 + 位置綁定 |
| **JWT-X5C** `jwt_payload_norm` 是否等於 `b64url_decode(JWS payload section)` | ❌ 兩個 hash (`jwt_signed_hash` / `jwt_payload_b64h`) 完全無關 | ✅ in-circuit b64url decode + byte-equality |
| **spec-check** 公開輸入型別漂移 | ❌ 只比 name | ✅ 同時比 name + type，並加入 `[FAIL] index N (foo): spec=X abi=Y` |
| **gate cost** | sdjwt 5,841 / jwt_x5c 20,879 | sdjwt 35,885 / jwt_x5c 136,440 |
| **public ABI** | 不變 | 不變（新 witness 都是 private） |
| **Rust / Swift verifier** | — | **無需更新**（公開輸入沒動） |

## 1. SD-JWT P0-5：In-Circuit Inclusion Proof

### 1.1 舊版（v3 / v3.1）的 gap
```text
issuer ECDSA-signs:  jwt_payload_hash = SHA256(payload bytes)  ← public input
                              │
                              │ 在 circuit 外計算，circuit 不知道 payload bytes
                              ▼
prover 自由給:        expected_disclosure_hashes[i]            ← private witness
                              │
                              ▼
circuit 算:           sd_root = SHA256(combined hashes)
                     out_disclosure_root = pedersen(count, hashes)
```
**攻擊面**：prover 可以挑任何一組 32-byte「假 disclosure hash」當成 issuer 給的，因為 circuit 從來沒看過 payload bytes。Pedersen disclosure_root 只能告訴 verifier「這組 hash 是哪一組」，但 verifier 還是要在 off-chain 比對 issuer manifest。

### 1.2 新版 in-circuit 閉環
新增三個 private witness：

| Witness | 型別 | 用途 |
|---|---|---|
| `jwt_payload_raw` | `[u8; 1024]` | issuer 真正簽的 payload bytes |
| `jwt_payload_raw_len` | `u32` | 有效 payload 長度 |
| `disclosure_positions` | `[u32; 8]` | 每個 disclosure hash 的 base64url 形式在 payload 中的起始 offset |

加入兩條新斷言：

1. `sha256_var(jwt_payload_raw, len) == jwt_payload_hash`
   → `jwt_payload_raw` 提供的 bytes **必定**是 issuer 簽的 payload。

2. 對每個 active 的 disclosure：把 `expected_disclosure_hashes[i]` 經 `b64url_encode_hash32` 轉成 43 bytes，逐 byte 對到 `jwt_payload_raw[disclosure_positions[i] .. +43]`。
   → 該 hash **必定**是 issuer 簽的 payload bytes 的子串。
   → bound check 同時用 `+43 ≤ JWT_PAYLOAD_LEN` 與 `+43 ≤ jwt_payload_raw_len`，所以 prover 不能把假的 b64 編碼塞到 zero-padding 區塊。

### 1.3 殘留 gap（保留 off-chain policy）
- **不檢查 JSON 結構**：circuit 不證明 hash 一定在 `_sd: [...]` 鍵下，只證明它在 payload bytes 裡某個位置。理論上 issuer 可以把 hash 字串 echo 到一個無關欄位來騙人，但那是 **issuer-side** 可以避免的（template control）。
- 因此 `out_disclosure_root` 仍然保留，verifier policy 仍可比對 manifest，這是「belt-and-braces」設定。

## 2. JWT-X5C P1-4：JWS Payload ↔ jwt_payload_norm Binding

### 2.1 舊版（v3.1）的 gap
v3.1 已經做到：
- `sha256_var(jwt_signing_input, jwt_signing_input_len) == jwt_signed_hash` （P0-4）
- `digest(jwt_payload_norm) == jwt_payload_b64h`

但這兩條 hash **沒有任何關係**。Prover 可以：
1. 拿一份 issuer 真實簽過的 `jwt_signing_input`（嚴格通過 P0-4）
2. 自己捏造一份不同 claim 的 `jwt_payload_norm`（只要 `SHA256` 對得上 verifier 給的 `jwt_payload_b64h`）

→ commit 用的 claim 與 issuer 簽的 claim 可以完全不同。verifier 必須在 off-chain 比對「`jwt_payload_norm` 是否等於 `b64url_decode(payload section of jwt_signing_input)`」。

### 2.2 新版 in-circuit 閉環
新增三個 private witness：

| Witness | 型別 | 用途 |
|---|---|---|
| `payload_b64_offset` | `u32` | payload section 在 signing input 中的起始 byte |
| `payload_b64_len` | `u32` | payload section 的 base64 char 長度 |
| `payload_norm_len` | `u32` | decode 後的 byte 長度 |

加入四條新斷言：

1. `payload_b64_offset + payload_b64_len ≤ jwt_signing_input_len`（window 在簽的範圍內）
2. `payload_b64_offset ≥ 1` 且 `jwt_signing_input[payload_b64_offset - 1] == '.'`（dot anchor，prover 不能把 offset 滑到 header b64）
3. `b64url_decode_into(jwt_signing_input, offset, b64_len, norm_len) -> [u8; 1024]`
4. 該結果與 `jwt_payload_norm` **逐 byte 相等**

→ 結合 P0-4：issuer 簽的 bytes 一定 b64url-decode 出 `jwt_payload_norm`，所以 `extract_claim_by_tag` 抽到的 claim 一定是 issuer 簽的。

### 2.3 成本與 budget
| 指標 | 舊 | 新 |
|---|---|---|
| ACIR opcodes | 20,879 | 136,440（+115k） |
| `gate_budget` (spec.toml) | 65,000 | 200,000 |
| Artifact size | 1.3 MB | 3.0 MB |
| Bytes/gate | 62 (C grade) | 23 (B grade) |

成本來自：1024-byte 輸出緩衝區的 b64 decode（~342 個 4-char group）+ 1024 次 byte 比對。Noir 的 dynamic indexing 是主要成本，這是必要代價。

## 3. spec-check 公開輸入型別檢查（P2-10 補強）

### 3.1 舊版
```python
if abi_names == spec_names:
    ok(...)
```
只比較名稱。下面這種改動會悄悄通過：

```diff
- public_input_types = ["[u8; 32]"]
+ public_input_types = ["[u8; 64]"]   # ← spec drift，但 abi name 沒動
```

### 3.2 新版
新增 `abi_type_to_str()` 把 ABI 的 nested dict canonicalize 成 spec 字串：

| ABI dict | 字串 |
|---|---|
| `{"kind":"field"}` | `Field` |
| `{"kind":"boolean"}` | `bool` |
| `{"kind":"integer","sign":"unsigned","width":128}` | `u128` |
| `{"kind":"array","length":32,"type":{...u8...}}` | `[u8; 32]` |
| 巢狀陣列 | `[[u8; 32]; 4]` |

並回傳精準的差異訊息：
```
[FAIL] passport_verifier: public-input type drift: index 0 (modulus_limbs): spec='[u32; 18]' abi='[u128; 18]'
```

## 4. In-Circuit vs Off-Chain Policy — 取捨指南

> **重點**：以下兩條都是有效的安全模型，差別在於「信任邊界放在哪」。如果 off-chain UX/cost 優勢明顯且 verifier 願意承擔該段 policy 責任，**off-chain 也是可採納的**。

### 4.1 In-Circuit 的好處（v3.2 採用）
- ✅ **零信任 verifier**：verifier 只需要驗 proof + 公開輸入，**不必額外執行任何 policy 程式**。
- ✅ **跨 verifier 一致**：所有 verifier 都看到同一條規則，不會因為某 verifier 忘了做檢查就漏掉。
- ✅ **legal-grade evidence**：proof 本身就涵蓋整條 chain（issuer-signed bytes → claim → commitment）。
- ✅ **適合**：高敏感場景（KYC、護照、金融）、verifier 是匿名第三方、需要可被法律或審計追溯。

### 4.2 Off-Chain Policy 的好處
- ✅ **gate 成本低**：以 JWT-X5C 為例，in-circuit 多了 ~115k gates → prove time / RAM / 行動裝置電量都會跳。off-chain 完全免費。
- ✅ **彈性**：off-chain 規則可以隨時調整，不必重編 circuit、重發 verifying key。
- ✅ **支援更複雜的語意**：JSON schema、JWT claim 完整解析、issuer template diff，這些 in-circuit 做都很貴或不可能。
- ✅ **適合**：verifier 是受信任的服務（自家 backend）、UX 對 prove time 敏感（行動端 < 1s 預算）、規則本身會頻繁變動（issuer manifest）。

### 4.3 建議採用對照表

| 場景 | 建議 |
|---|---|
| 護照 + age proof（passport_adapter / openac_show） | **必 in-circuit**（已是 v3.x 設計） |
| 校園 SD-JWT + 學歷揭露 | **in-circuit**（v3.2 已閉環）；若 prove time 限制 < 5s 且 verifier 是自家服務，可考慮退回 off-chain manifest 模式並下調 gate |
| 企業 OIDC（jwt_x5c）+ 權限 claim | 取決於 verifier：**B2B 自家** → off-chain 也可；**第三方 SSO sink** → in-circuit |
| 短期一次性票券 / 行銷碼 | **off-chain 為主**：UX 重於密碼學嚴謹度 |
| 跨組織數位身分（W3C VC） | **必 in-circuit**：法律可追溯性 > 一切 |

### 4.4 採用 off-chain 時必備的 verifier checklist
若團隊評估後決定走 off-chain（例如 SD-JWT 場景），verifier 端**至少**要做：

1. 對照 `jwt_payload_hash`（public input）與 issuer 公布的 manifest 雜湊；
2. 對 `out_disclosure_root` 與 issuer manifest 中允許的 disclosure 集合比對；
3. 取得 issuer-signed payload bytes，自行 base64url-decode 並對照 commit 中宣稱的欄位；
4. 紀錄 verifier-side 的這份比對在 audit log，因為 proof 本身不再是 self-contained evidence。

→ 上述任一條漏掉就退回到 v3 / v3.1 的 gap，等於沒修。

## 5. 對 iOS App / Rust mopro / Swift Verifier 的影響

| 元件 | 影響 |
|---|---|
| **公開輸入 ABI** | **不變** —— 新增 witness 都 private。 |
| **mopro-binding (Rust)** | 不需要改 verifier（仍 strict ABI v3 layout）；但**測試 fixture / 行動端 prepare flow** 必須準備 `jwt_payload_raw` / `jwt_payload_raw_len` / `disclosure_positions`（SDJWT）以及 `payload_b64_offset` / `payload_b64_len` / `payload_norm_len`（JWT-X5C）。 |
| **iOS app 端的 prepare flow** | 必須能拿到 issuer-signed raw payload bytes（SD-JWT 通常是 JWS 第二段 base64url 解開後的 JSON bytes；JWT-X5C 也是同樣）。已是現行流程，只是先前沒餵進 circuit。 |
| **Swift OpenPassportSwift verifier** | **無需改動**；既有 v3 strict ABI 驗證仍 21/21 pass。 |
| **prove time** | sdjwt + jwt_x5c 預計 +30-50%（依裝置而定）。如果觀察到行動端體感變差，可優先評估 4.3 的 off-chain 退回方案。 |
