## 2. New finding — noir_rs / circuit toolchain mismatch (P0 blocker)

`make bench-prove-verify` ran but failed at the first `generate_noir_proof`
call:

```
CircuitError: Incompatible noir_version
  '1.0.0-beta.19+74d6be658e1ad252f87943292ba09bdd4da80bd4'.
This binding supports '1.0.0-beta.8.x' circuit artifacts
(noir_rs v1.0.0-beta.8-3).
```

Where this comes from:
- Circuits compile with `nargo 1.0.0-beta.19` (pinned in CLAUDE.md).
- `mopro-binding/Cargo.toml` resolves `noir_rs` to v1.0.0-beta.8-3 which
  only accepts beta.8 ACIR artifacts.

What this means:
- `bench_prove_disclosure`, `bench_verify_disclosure`, and
  `bench_proof_size_bytes` all error out **before** doing real work.
- `bench_memory_usage` "passes" only because the assertion `after < 2048`
  trivially holds when no prove ever runs (RSS delta = 1 MB is meaningless).
- The PERF-1 timestamp in `/tmp/bench-prove-verify.log` is `386.542µs` —
  that is the time-to-error, not real prove time.

Resolution paths (P0 for next session, in preference order):

1. **Upgrade mopro-binding deps** to a noir_rs / barretenberg release
   that accepts beta.19 artifacts. Check upstream `mopro` and `noir_rs`
   for a tagged version compatible with `1.0.0-beta.19`. This is the
   recommended path — keeps recent security/feature fixes in circuits.
2. **Pin nargo to beta.8 and recompile circuits.** Loses recent work
   (Path A v3, openac_core security fixes). Not recommended.

Documented in `benchmark/expected/baseline.toml` `[performance]` section.

**Discovery during backfill**: composite_show and x509_show were
*pre-populated* per the 4/26 handoff §3.3, but the populated values are
for an older v3 main() signature. Running `nargo execute` on them now
fails with `Expected argument 'a_attr_hi'` (composite) or
`Expected argument 'link_scope'` (x509) — fields that the current
main() expects but the toml doesn't have. So those two also count as
"needs backfill" despite having content.

Backfill recipe for the remaining 8:

1. Open the circuit's `src/main.nr`. Find the inline `#[test]` whose
   inputs you want to mirror (usually `test_valid_*` or first positive).
2. For circuits that involve **only sha256 / byte arrays** (the 4 done
   today), you can compute outputs offline in Python and paste into
   `Prover.toml`. Took ~10 min per circuit.
3. For circuits with **Pedersen-hash outputs** (`commit_attributes_v3`,
   `pedersen_hash([...])`, all v3 main path circuits), the cleanest
   path is to add a temporary `println!` to the test, run
   `nargo test --show-output --package <name>`, capture the dumped
   field values, paste into Prover.toml, then revert the `println!`.
   The pre-existing comments in `composite_show/Prover.toml` and
   `x509_show/Prover.toml` describe this method ("Public outputs were
   generated by running test_valid_show with println() instrumentation;
   regenerate by reintroducing a dump helper if these inputs ever
   change") — that's the canonical recipe.
4. For RSA-heavy circuits (`passport_verifier`, `passport_adapter`,
   `data_integrity`), the inputs are 18 × u128 limbs that are hard to
   write by hand. Same dump-via-println approach, but expect ~1-2 hours
   per circuit because the inputs are large and the cross-checking is
   tedious.

Estimated remaining work: **4-6 hours** for the 8 mismatched/empty
fixtures. After that, `make bench-execute` produces witness-gen times
for all 12 — useful for relative prove-time ranking even before the
noir_rs upgrade lands.
